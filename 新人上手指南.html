VGAME DESIGN HARNESS

Vgame 驾驭工程新人使用手册

从第一次配置路径、启动 Codex，到 Git/SVN 提交、知识图谱维护和九步策划交付，都集中在这一份手册中。仓库保存规则与知识，不保存任何成员的本地盘符。

中文项目规范 真实配置默认只读 三源知识图谱 9 步完整策划交付 Vgame 专项 Skill 路由

第一次安装

先分清三类位置：SVN 游戏工程、包含驾驭工程的 Git 策划仓库，以及个人输入与输出目录。

变量内容权限
${VGAME_ROOT}真实 Excel、代码和 Unity 资源默认只读，写入需明确授权
${VGAME_DESIGN_ROOT}Design、Proposal、Task、Skill、整合图谱和驾驭工程按任务维护并提交 Git
${VGAME_HARNESS_ROOT}Git 仓库内的中文安装、更新、检查、启动按钮按任务维护并提交 Git
${VGAME_SOURCE_DOCS_ROOT}英雄设计、经济框架、战斗框架等原始资料只读输入
${VGAME_OUTPUT_ROOT}量化表、审查报告和模拟结果个人输出

操作顺序

  1. 准备工具
    安装 Git、TortoiseSVN、Python 3.10+、Codex，并确认公司 Git SSH 权限。
  2. 克隆策划仓库
    用 Git 克隆 Vgame_design，进入其中的 驾驭工程目录。
  3. 配置路径
    双击 配置本机路径.bat，填写 SVN 根目录、源文档目录和个人输出目录。Git 根目录由脚本自动确定。
  4. 运行安装
    双击 一键安装.bat，等待依赖、仓库、Skill 和图谱检查完成。
  5. 确认结果
    正常结果为 PASS (FAIL: 0, WARN: 0) 和 Setup complete。
  6. 启动图谱和 Codex
    双击 启动知识图谱.bat，然后用 Codex 打开 ${VGAME_DESIGN_ROOT}。
路径不会上传：本机配置保存在 %LOCALAPPDATA%\Vgame\design-harness.env.bat 和 Git 忽略的 local.env.bat。每个人可以使用不同盘符。

每天怎么启动

  1. 更新真实项目
    右键 ${VGAME_ROOT} 或本次涉及目录，执行 SVN Update。
  2. 更新策划仓库
    双击 更新驾驭工程.bat。存在本地修改时脚本会停止保护现场。
  3. 运行检查
    双击 检查驾驭工程.bat，阅读所有 FAIL 和 WARN。
  4. 按需启动 Dashboard
    浏览器通常打开 http://127.0.0.1:5173，以启动窗口显示地址为准。
  5. 开始任务
    在 Codex 中明确任务范围、是否只读、是否允许修改真实配置。
更新提示本地修改时：进入 ${VGAME_DESIGN_ROOT} 执行 git status，不要删除或还原不认识的文件。

在全新对话中怎么使用

新对话不会继承旧聊天记忆，但会读取 Vgame_design Git 仓库根目录的 AGENTS.md，并通过本机路径连接策划总控、项目 Skill、知识图谱和 SVN 真实工程。

  1. 新建 Codex 对话
    工作目录选择 ${VGAME_DESIGN_ROOT}，不再从 SVN 工程目录启动。
  2. 自动加载项目入口
    根目录 AGENTS.md 会读取本机路径配置、${VGAME_ROOT} 和 vgame-core-understanding。
  3. 直接说明任务
    写清任务、范围、是否只读、是否允许修改真实配置，不需要重新介绍整个项目。
  4. 观察加载结果
    AI 应先确认项目入口和相关专项 Skill，再读取真实证据。

快速开始

直接说功能、目标和限制即可。总控先解析项目、控制模式、交付档位、风险档位和写入权限，再挂载 Vgame 项目适配层。

完整交付示例：“为 Vgame 设计一个新的周期 Boss 挑战，交付 DOC_UI，按 HIGH 风险执行，先做需求契约，不修改真实配置。”
只读分析示例：“CONFIG_REVIEW：分析资源副本金钱本什么时候开放，列出表、字段、奖励引用和风险，不修改任何文件。”

启动顺序

  1. 总控解析任务
    识别 ADVICE / DISCOVERY / DESIGN / CONFIG_REVIEW / CONFIG_SYNC / FULL_DELIVERY。
  2. 挂载 Vgame 适配层
    读取 skills/config.toml、vgame-core-understanding、项目画像、证据地图、配置索引和交付门禁。
  3. 确认项目证据
    只做高层讨论时不乱读 Excel；需要文件结论时按证据优先级回到真实源。
  4. 选择工作流深度
    建议和只读审查走最小路径；完整功能才启动九步交付。
  5. 人工门禁
    需求契约、澄清、规则冻结、验收就绪和最终验收均不可由 Agent 自动代签。

控制模式、交付档位与风险档位

三组参数解决不同问题：控制模式决定“做多深”，交付档位决定“交什么文件”，风险档位决定“评审多严格”。

控制模式

ADVICE只给建议、风险或方案，不生成文件。
DISCOVERY建立项目画像、证据地图和待确认清单。
DESIGN需求澄清、范围契约和功能规则设计。
CONFIG_REVIEW只读检查 Excel、JSON、ID 和引用链。
CONFIG_SYNC用户明确授权后同步配置与产物。
FULL_DELIVERY九步流程、正式 UI/Excel、验收和交付。

交付档位

档位正式交付物适用场景
ADVICE对话结论方向咨询、风险判断
DOC_ONLYExcel 策划案 + 追踪矩阵 + 验收用例规则文档与配置策划
UI_ONLYHTML + 页面/组件 PNG + UI Manifest只做界面规格和视觉验证
DOC_UIExcel + HTML + PNG + Manifest，且 Excel 嵌图默认完整策划交付
IMPLEMENTATION实际修改 + 验证证据 + 追踪与验收代码或配置落地

风险档位

LOW 小范围

单模块、小调整、无复杂数值或跨系统影响。专项角色与总控自检。

MEDIUM 多产物

涉及配置、Excel、UI、周期奖励或多个系统。必须做一次一致性评审。

HIGH 核心风险

经济投放、付费、复杂战斗、版本核心或线上变更。必须独立评审和成品复核。

工作流总览

Vgame 完整交付由通用总控编排，先执行项目适配 Step 0，再进入九步功能交付。所有下游角色读取同一份需求契约和项目证据，避免各自理解一套规则。

Step 0 · 项目适配：game-planning-control → skills/config.toml → vgame-core-understanding → 项目画像 / 证据地图 / 配置索引 / 交付门禁。

回退原则

发现问题的位置回退规则
核心设计范围变化回需求契约；规则细节问题留在核心设计。
专项设计回对应专项；核心规则变化则回步骤 4。
UI / Excel内容问题回设计，纯视觉或排版问题留在本阶段。
验收 / 交付映射缺失回对应产物；规则不可验收回评审或设计。
同一阶段最多自动返工两轮。超过后必须标记 blocked，列出分歧、已尝试方案和需要用户决策的问题。

Git 与 SVN 怎么操作

Git 管策划知识，SVN 管真实游戏项目。两种 Commit 的含义不同，不能混用。

Git：策划仓库

git status
git pull --ff-only origin master
git add "具体文件路径"
git diff --cached
git commit -m "docs(guide): update workflow"
git push origin master
  • add 选择本次提交文件。
  • commit 只在本机创建记录。
  • push 才会上传公司 Git。

SVN：真实项目

  1. 开工前执行 SVN Update。
  2. 完成后执行 Check for modifications。
  3. 新文件先执行 SVN Add。
  4. SVN Commit 时只勾选本次确认文件。

SVN Commit 会直接上传服务器。发生 Conflicted 时停止提交并确认合并方案。

禁止操作：不要随意使用 git add .、git push --force、git reset --hard、SVN 根目录全选提交或未经确认的 Revert。

跨仓任务

真实配置与 Skill 同时变化时，先更新两边，在 Git 记录设计和影响范围，获授权后修改 SVN，验证真实结果并提交 SVN，再更新 CROSS-REPO-CHANGES.md、Task 和 Skill，最后推送 Git。

常见问题

只想问一个配置问题，也必须写 Design 吗？

不需要。只读事实查询直接走图谱和专项 skill，输出证据链即可。只有形成新规则或正式变更时才进入 Design / Proposal / Task。

知识图谱和 Skill 冲突时听谁的？

两者都不是最终真源。回到真实 Excel、代码和已确认设计核实；核实后修正图谱生成逻辑或回写 Skill。

为什么默认不允许修改真实配置？

分析和写入的风险不同。未经授权写表可能覆盖公式、破坏引用或制造版本冲突，因此真实写入必须由用户明确要求。

新增配置表后做什么？

先确认表已注册到 __tables__.xlsx，再全量更新图谱，并把表作用和关联补入相应项目 Skill。

换电脑或换目录后怎么用？

双击 驾驭工程/配置本机路径.bat 重新选择目录即可。仓库文档、脚本和 JSON 不依赖任何成员的盘符。

安装出现 WARN 怎么办？

先看最终统计。FAIL 必须处理；WARN 必须阅读原因并确认是否影响当前任务。正常首次安装应为 FAIL: 0, WARN: 0。

什么时候需要更新知识图谱？

新增、删除或修改配置表字段，修改 __tables__.xlsx，客户端结构大改，或新增重要 Design、Proposal、Task、Skill 时更新完整图谱。只改普通数值通常不需要。

AI 找不到英雄设计或经济框架怎么办？

重新运行 配置本机路径.bat，确认 Source design documents root 指向源策划文档实际所在目录。

更新驾驭工程提示本地有修改怎么办？

执行 git status 查看清单。不要删除不认识的修改；确认归属后提交，或交给原修改者处理。

Skill 已加入，为什么状态脚本还不能运行？

Skill 定义了流程协议和命令契约，但对应 Python 文件没有随当前仓库出现。手册会明确标为“待接入”，避免把协议描述误当成已实现能力。

新人安全红线

  1. 真实项目默认只读
    未获明确授权，不修改真实 Excel、代码和 Unity 资源。
  2. 提交前看清清单
    Git 和 SVN 都只提交本次确认文件。
  3. 保护他人现场
    不提交、不删除、不还原不认识的修改。
  4. 个人路径不上库
    local.env.bat、个人输出和缓存不得加入版本库。
  5. 源 Excel 才是真源
    不直接修改 server_json 等生成物替代源表。
  6. 不强行覆盖冲突
    不使用强制 Push、硬重置或未经确认的 SVN 冲突覆盖。
  7. 门禁结果必须解释
    有 FAIL 不交付；有 WARN 必须说明处理结论。
