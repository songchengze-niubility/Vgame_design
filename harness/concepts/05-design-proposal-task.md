# 设计 -> 提案 -> 任务流程

> 先设计，后提案，再拆任务。不是官僚主义，是为了少返工。

## 流程总览

```text
Idea
  -> Design      design/
  -> Proposal    proposals/
  -> Task        tasks/
  -> Config/Code ${VGAME_ROOT}
```

## Design

用于回答“该不该做、怎么设计、影响什么”。

产物：`design/<主题>.md`

适用：

- 新玩法、新系统、新角色定位。
- 跨表配置链变动。
- 影响经济、战斗、开放节奏、版本节奏。

## Proposal

用于回答“怎么落地、怎么验收、风险怎么控”。

产物：`proposals/<主题>-YYYY-MM-DD.md`

要求：

- 引用 Design Doc。
- 列出影响表、代码、skill、文档。
- 写明验证方式。

## Task

用于回答“谁做什么、做到哪一步算完成”。

产物：`tasks/<主题>-YYYY-MM-DD.md`

要求：

- 引用 Proposal。
- 拆分可验证子任务。
- 跟踪阻塞和验收。

## 快捷通道

简单只读分析、单点文档修正、已明确的小配置说明，可以直接写 Task 或审查记录。

不能走快捷通道：

- 新增系统。
- 跨多个配置表联动。
- 影响经济总量或战斗强度。
- 影响开放节奏、商业化、安全或版本发布。
