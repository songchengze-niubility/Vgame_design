# Proposal: 爬塔多编队挑战

**完整策划案：** [爬塔多编队策划案](../output/爬塔多编队/爬塔多编队策划案.md)  
**Excel：** [爬塔多编队策划案.xlsx](../output/爬塔多编队/爬塔多编队策划案.xlsx)

## 1. 目标

在现有 100 层爬塔上引入多编队挑战，使中后段玩法从单队纵向强度验证扩展为角色池宽度与多队构筑验证。

## 2. 核心方案

- 所需编队数：`min(最大编队数, 基础编队数 + floor((楼层-1)/X))`。
- 草案基准：`X=20`、基础 1 队、最大 5 队、每队 5 人、跨队角色不可重复。
- 每支编队独立完成一场子战斗；任一失败则整层失败；全部胜利后沿用现有通关与奖励链。
- 本期不修改奖励数值，扫荡继续沿用现有资格。

## 3. 影响范围

| 模块 | 影响 | 说明 |
|---|---|---|
| UI | 中 | 塔主界面、多编队页、战斗序号、汇总结算。 |
| 配置 | 中 | 新增多编队规则与子战斗配置，注册 Luban 表。 |
| 客户端 | 中 | 单阵容数组升级为阵容组，增加塔专用阵容存储。 |
| 服务端 | 高 | 多阵容校验、子战斗实例、重连与幂等结算。 |
| 奖励经济 | 低 | 不改数量，仅改为全队胜利后统一触发。 |

## 4. 评审结论要求

评审需确认：多队战斗形态、角色是否可重复、`X` 与最大队数、失败是否整层重打、奖励是否保持不变。确认后再拆正式任务与配置实施；未经授权不修改 `${VGAME_ROOT}` 下真实表和代码。

## 5. 验收

- [ ] 参数边界与最大队数计算正确。
- [ ] 不完整或重复角色编队无法开始挑战。
- [ ] 任一子战斗失败不推进、不发奖。
- [ ] 全部胜利只推进一次、只发奖一次。
- [ ] 功能开关可回退单队流程。
- [ ] Excel、规则和验收用例完成追踪。
