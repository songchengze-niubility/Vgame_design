## 背景与目标
### 背景
月卡是二次元卡牌游戏中最稳定的小额付费 ARPU 来源，通过"首购立即奖励 + 30 天每日返还"的模式建立付费习惯。卡面包括 Price、PayProductId、firstReward（首购奖励）、dailyReward（每日奖励）及 PowerLimitUP（体力上限提升）等权益。

### 目标
- 支持多档月卡（如基础/高级）配置
- 首购立即发放 firstReward
- 购买后 30 天内每日登录可领取 dailyReward
- 体力上限提升 PowerLimitUP 权益生效于有效期内
- FirstRewardShow + DailyRewardShow 用于预览展示

### 非目标
- 不设计月卡叠加（同一类型不支持多张叠加）
- 不处理月卡过期后体力上限回退的容错细节（由体力系统处理）
- 不设计月卡续费优惠

## 核心规则
### R-01: 购买月卡
- **触发条件**：玩家在月卡界面点击"购买"按钮
- **前置限制**：玩家未持有对应类型月卡 或 月卡有效期 ≤ 某阈值（允许续购）；持有对应 PayProduct
- **操作流程**：拉起支付 → 支付成功 → 激活月卡 → 发放 firstReward → 月卡剩余天数设为 30（或续期累加）
- **状态变化**：月卡激活状态 = true；剩余天数 = 30（或 30 + 原剩余天数）
- **资源变化**：-RMB；+firstReward 奖励
- **UI 反馈**：购买成功动画 → 月卡卡片亮起 → 首购奖励飞入 → 剩余天数展示
- **异常边界**：支付失败 → 不激活不发奖；续购天数上限（如 90 天）
- **落地点**：MonthCard.xlsx

### R-02: 每日奖励领取
- **触发条件**：持有有效月卡的玩家每日登录
- **前置限制**：月卡剩余天数 > 0；当日未领取每日奖励
- **操作流程**：读取 MonthCard.xlsx 中 dailyReward → 发放奖励 → 标记当日已领 → 剩余天数 -1（或仅计时到期）
- **状态变化**：当日领取标记 = true
- **资源变化**：+dailyReward 奖励
- **UI 反馈**：登录弹窗/月卡界面"今日可领" → 领取后显示"已领取" → 剩余天数更新
- **异常边界**：跨天未登录 → 不补发之前未领奖励；连续 N 天未登录不扣减剩余天数（天数按自然日计算，不论登录与否）
- **落地点**：MonthCard.xlsx

### R-03: 体力上限提升
- **触发条件**：玩家持有有效月卡
- **前置限制**：MonthCard.xlsx 中 PowerLimitUP 字段 > 0
- **操作流程**：月卡激活时将 PowerLimitUP 追加到玩家体力上限属性；月卡过期时回退
- **状态变化**：体力上限 = 基础上限 + PowerLimitUP
- **资源变化**：无（仅上限变化，不影响当前体力值）
- **UI 反馈**：体力条上限数字增加 → 面板标注"月卡加成 +X"
- **异常边界**：过期时当前体力 > 基础上限 → 保留当前体力不削减，仅上限回退
- **落地点**：MonthCard.xlsx

### R-04: 月卡过期
- **触发条件**：月卡剩余天数降至 0（或自然时间到达激活日 + 30 天）
- **前置限制**：无
- **操作流程**：月卡激活状态设为 false → 回退 PowerLimitUP → 界面隐藏每日奖励入口
- **状态变化**：月卡状态 = 过期
- **资源变化**：体力上限回退（见 R-03）
- **UI 反馈**：月卡卡片变灰 → 提示"已过期，续费享受特权" → 体力上限数字还原
- **异常边界**：剩余天数计算精度（按自然日 5:00 刷新处理）
- **落地点**：MonthCard.xlsx

### R-05: 奖励预览展示
- **触发条件**：月卡界面打开（未购买前 / 购买后）
- **前置限制**：无
- **操作流程**：读取 FirstRewardShow + DailyRewardShow → 拼接预览列表展示
- **状态变化**：无
- **资源变化**：无
- **UI 反馈**：月卡卡片上展示"立即获得"（首购奖励列表）+ "每日获得"（每日奖励摘要）
- **异常边界**：预览字段为空 → 不展示对应区块
- **落地点**：MonthCard.xlsx

### R-06: 多档月卡共存
- **触发条件**：玩家同时持有基础月卡和高级月卡（若存在）
- **前置限制**：配置中存在多个月卡产品
- **操作流程**：各月卡独立计时、独立权益叠加 → 每日奖励分别领取 → 体力上限加成累加
- **状态变化**：各月卡独立状态与剩余天数
- **资源变化**：各月卡独立的 dailyReward
- **UI 反馈**：多张月卡卡片水平/垂直排列，分别展示剩余天数与领取状态
- **异常边界**：同类型月卡只允许一张（不允许重复购买同类型）
- **落地点**：MonthCard.xlsx

## 技术方案
### 架构影响
- 月卡剩余天数按自然日计算（每日 5:00 标记消耗一天），不需要玩家登录触发扣天
- 体力上限属性需支持多来源叠加与回退，月卡作为来源之一注册
- PayProduct 购买与月卡激活通过支付回调异步衔接

### 数据变更
| 表 | 变更类型 | 说明 |
|----|----------|------|
| MonthCard.xlsx | 配置 | Price, PayProductId, firstReward, dailyReward, PowerLimitUP, FirstRewardShow, DailyRewardShow, CostItem |

### 接口变更
- `MonthCardGetInfo`：出参 CardList[] (CardId, RemainDays, IsDailyClaimed, IsActive)
- `MonthCardPurchase`：入参 CardId + PayProductId；出参 FirstRewardItems[] + NewCardInfo
- `MonthCardDailyClaim`：入参 CardId；出参 RewardItems[]
- 体力上限变更通过属性变更事件推送至体力模块

## 风险评估
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 每日奖励未领但天数照扣 → 玩家损失 | 高 | 中 | 产品定义"按自然日，不登录不领不补"并明确告知 |
| 体力上限回退时数值异常 | 低 | 中 | 属性系统支持来源追溯，回退精准 |
| 月卡到期提醒不及时导致断卡 | 中 | 低 | 剩余 ≤ 3 天时弹窗提醒续费 |
| 首购奖励配置过高导致刷首充 | 中 | 中 | firstReward 价值控制在合理范围 |
| PayProduct 掉单 | 低 | 高 | 支付回调重试 + 对账补发 |
