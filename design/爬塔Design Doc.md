## 背景与目标
### 背景
爬塔是 Vgame 中 100 层的渐进式战斗挑战。每层设置独立怪物编队和数值配置，按楼层类型（小层 / 中层 / 大层）区分难度系数（0.05 / 0.2 / 0.4）。怪物等级从 20 起步，随楼层递增至约 200 级。前期楼层为一次性首通，后期楼层引入周期性刷新机制，收入来自首通奖励 + 周期性奖励 + 排名奖励三线。

### 目标
- 提供 100 层长期渐进式目标，预期约 60 天全通
- 通过楼层类型差异化产生难度波动节奏（平缓 → 跃升 → Boss 关）
- 三线奖励并行驱动：首通一次性获取 + 周期性复刷 + 排名竞争
- 验证玩家的装备深度和阵容策略

### 非目标
- 非纯竞速（排名非核心、奖励辅助性）
- 非多人合作爬塔
- 非无限层（严格 100 层封顶）

## 核心规则
### R-01: 楼层结构与类型
- **触发条件**：玩家进入爬塔主界面
- **前置限制**：楼层配置表加载完毕（`爬塔难度线` sheet + TowerTrial 相关表）
- **操作流程**：100 层分为三种类型 —— 小层（coefficient 0.05）：普通怪物编队，难度增幅平缓；中层（coefficient 0.2）：含精英怪，难度跃升明显；大层（coefficient 0.4）：Boss 层，大关考验
- **状态变化**：楼层类型在配置时固定，运行时不变
- **资源变化**：无
- **UI 反馈**：楼层列表/地图界面用不同颜色或图标区分三种类型；大层有特殊 Boss 标识
- **异常边界**：如配置表中某层缺少类型字段，默认按小层处理并上报异常日志
- **落地点**：楼层配置解析 → UI 渲染

### R-02: 怪物等级渐进
- **触发条件**：进入新楼层
- **前置限制**：怪物初始等级 20（第 1 层），递增率约 1.8/层
- **操作流程**：计算公式 `level = 20 + floor * 1.8`（approx），第 100 层达到约 200 级；每层怪物属性按等级 + 类型系数缩放
- **状态变化**：怪物数据按公式生成
- **资源变化**：无
- **UI 反馈**：楼层界面显示推荐等级 / 推荐战力（可选）
- **异常边界**：浮点取整逻辑统一（向下取整/四舍五入由配置决定）；配置改动后已生成数据不回溯
- **落地点**：怪物生成器

### R-03: 一次性首通奖励
- **触发条件**：玩家首次通关某一楼层
- **前置限制**：首通奖励配置从 TowerFirstReward.xlsx 读取；首通记录为每层独立标记
- **操作流程**：通关结算时判断首通状态 → 首次通关发放 TowerFirstReward 奖励 → 标记该层首通已完成；同层再次通关不再发放
- **状态变化**：楼层首通标记 `FirstClear → Cleared`
- **资源变化**：一次性发放装备、货币、材料
- **UI 反馈**：结算界面首通宝箱动画；楼层列表中已首通的楼层显示勾选标记
- **异常边界**：首通检查幂等；若发放时背包满则补发邮件
- **落地点**：结算 → TowerFirstReward.xlsx → 背包/邮件

### R-04: 周期性刷新与复刷奖励
- **触发条件**：后期楼层在配置的刷新周期到期
- **前置限制**：刷新周期和覆盖楼层范围由 TowerReward.xlsx 定义；早期楼层不刷新（永久一次性）
- **操作流程**：达到刷新周期时，指定范围的楼层通关状态重置（重新锁定），玩家可再次挑战并获取 TowerReward 奖励；重置后原首通标记转为 "周期已清"
- **状态变化**：楼层状态 `Cleared → Active`（可再次挑战）
- **资源变化**：周期奖励按 TowerReward.xlsx 发放（奖励幅度低于首通）
- **UI 反馈**：已刷新楼层显示 "NEW" 角标或特殊边框；刷新倒计时提示
- **异常边界**：刷新时若玩家正在该楼层战斗中 —— 当前战斗不受影响，结算后该层变为可重新挑战状态；刷新时的断线不丢奖励
- **落地点**：定时任务 → 楼层状态管理 → TowerReward.xlsx

### R-05: 楼层解锁逻辑
- **触发条件**：玩家通关当前最高已解锁楼层
- **前置限制**：楼层必须按顺序解锁（1→2→3→...→100）
- **操作流程**：通关 N 层 → 自动解锁 N+1 层；首次进入爬塔时仅第 1 层解锁；不可跳层挑战
- **状态变化**：楼层 N+1 `Locked → Unlocked`
- **资源变化**：无
- **UI 反馈**：新解锁楼层震动/光效提示；解锁路径动画（上一层 → 下一层连线点亮）
- **异常边界**：若因 BUG 跳层战斗，服务器校验解锁链，未解锁楼层返回 "楼层未解锁" 错误码；已通关楼层仍可重复挑战（无首通奖励）
- **落地点**：楼层解锁控制器

### R-06: 排名奖励
- **触发条件**：排名结算周期到达
- **前置限制**：排名依据为玩家已通关的最高楼层 + 通关速度（同层比时间）；TowerRankReward.xlsx 定义奖励档位
- **操作流程**：周期结算时生成排名快照 → 按档位匹配奖励 → 邮件发放
- **状态变化**：排名快照写入数据库
- **资源变化**：发放排名奖励
- **UI 反馈**：爬塔界面显示当前排名定位和预计奖励档位；结算邮件
- **异常边界**：排名并列时的处理 —— 按通关时间戳先后排序（先通者排前）
- **落地点**：排名系统 + TowerRankReward.xlsx

### R-07: 预期推进节奏（60 天全通）
- **触发条件**：数值设计阶段的宏观验证
- **前置限制**：以每日固定推进楼层数 + 战力自然增长为假设
- **操作流程**：前 30 天 —— 玩家战力快速增长期，可日均推进 2-3 层（第 1-60 层）；中 20 天 —— 增长放缓，日均 1-2 层（第 61-85 层）；后 10 天 —— 最后 Boss 层，日均 1 层，需装备/策略深度支撑（第 86-100 层）
- **状态变化**：不适用（设计期分析）
- **资源变化**：不适用
- **UI 反馈**：不适用
- **异常边界**：若玩家推进速度明显偏离预期（如 10 天全通），可能表明数值平衡有误，需复盘等级/装备曲线
- **落地点**：数值设计文档

## 技术方案
### 架构影响
- 新增 `TowerMode` 运行模式
- 楼层状态管理模块（解锁、首通、周期刷新）
- 三套奖励配置表接入：TowerFirstReward、TowerReward、TowerRankReward

### 数据变更
| 表 | 变更类型 | 说明 |
|----|----------|------|
| 爬塔难度线 sheet（Vgame基本战斗框架.xlsx） | 已有 | 楼层等级/系数配置 |
| TowerFirstReward.xlsx | 新增 | 首通一次性奖励 |
| TowerReward.xlsx | 新增 | 周期性复刷奖励 |
| TowerRankReward.xlsx | 新增 | 排名奖励 |
| player_tower_progress | 新增 | 玩家楼层进度（当前层、首通标记、周期状态、最高层） |

### 接口变更
- 楼层列表查询：返回各层状态（锁定/已解锁/已首通/周期可用）
- 进入楼层：校验解锁 + 返回战斗配置
- 楼层结算：判断首通 + 发放对应奖励（首通 or 周期 or 无）
- 排名查询 / 结算领取

## 风险评估
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 后期楼层数值曲线不合理（过难/过易） | 中 | 中 | 上线前数值验证 + 前两周监控数据 + 热更调整 |
| 周期刷新逻辑 BUG 导致奖励重复发放 | 低 | 高 | 发放幂等 + 数据库约束 + 灰度测试 |
| 排名对高付费玩家偏斜过大 | 中 | 低 | 排名权重以楼层为主（非 PVP 强度）、低权重时间 |
