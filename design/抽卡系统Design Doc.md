## 背景与目标
### 背景
抽卡系统是游戏的核心商业化管道，通过角色卡池与专武卡池双线并行，驱动玩家的钻石消耗、票券收集及付费转化。系统基于概率权重表（DrawCardWeight）实现稀有度分级产出，并引入软保底（soft pity）机制在指定抽数后提升 SSR 权重。

### 目标
- 提供角色与专武两套独立抽卡池，支持限时/常驻票券分流
- 60 抽硬保底，约 35 抽为期望 SSR 出货点
- 重复转化：英雄重复产出 FirstReward + RepeatReward，武器重复产出 WeaponDuplicateReward
- 前 7 天通过各系统投放 ~70+ 限时角色票、~20+ 限时专武票

### 非目标
- 不处理跨池保底继承（当前独立计算）
- 不设计付费专属限定池（仅区分票券类型）
- 不处理保底计数器跨期重置逻辑（由运营活动表控制）

## 核心规则
### R-01: 角色单抽
- **触发条件**：玩家在角色卡池界面点击"抽一次"
- **前置限制**：主线进度 ≥ 1-1；持有 ≥ 1 张角色票券（普通 106 / 限时 111）或钻石 ≥ 150
- **操作流程**：客户端扣除资源 → 服务端查询 DrawCardPool 与 DrawCardWeight 按权重随机出卡 → 角色入库 → 发奖
- **状态变化**：保底计数器 +1；若 SSR 则计数器归零
- **资源变化**：-150 钻石 或 -1 对应票券；+1 角色
- **UI 反馈**：抽卡动画 → 展示角色卡面与品质 → 更新保底剩余次数
- **异常边界**：资源不足弹窗引导氪金/获取票券；网络超时补发
- **落地点**：DrawCard.xlsx, DrawCardPool.xlsx, DrawCardWeight.xlsx

### R-02: 角色十连抽
- **触发条件**：玩家在角色卡池界面点击"十连抽"
- **前置限制**：主线进度 ≥ 1-1；持有 ≥ 10 张角色票券 或钻石 ≥ 1500
- **操作流程**：批量执行 10 次单抽逻辑，合并展示结果
- **状态变化**：保底计数器 +10；每命中 SSR 即归零
- **资源变化**：-10 票券 / -1500 钻石；+10 角色
- **UI 反馈**：十连跳过/逐张展示动画 → 汇总面板展示所有结果 → 保底更新
- **异常边界**：资源扣除前校验总资源充足；若中途断线使用日志补发
- **落地点**：DrawCard.xlsx

### R-03: SSR 软保底
- **触发条件**：保底计数器达到软保底阈值（约 50-60 区间）
- **前置限制**：当次抽取不是 SSR
- **操作流程**：权重表中 SSR 权重阶梯提升（DrawCardWeight 读取对应索引权重列）
- **状态变化**：保底计数器在当前十连/单抽结束后继续累加
- **资源变化**：无额外
- **UI 反馈**：提示"保底即将触发"（弱引导，不暴露具体数值）
- **异常边界**：权重表缺失对应行时 fallback 默认 SSR 权重
- **落地点**：DrawCardWeight.xlsx

### R-04: 硬保底（SSR 必出）
- **触发条件**：距上次 SSR 已累计抽数 = 59（即第 60 抽必为 SSR）
- **前置限制**：第 60 抽执行时
- **操作流程**：强制本次结果为 SSR（跳过权重表，从池内 SSR 列表按等权重抽取）
- **状态变化**：保底计数器归零
- **资源变化**：同正常单抽扣除
- **UI 反馈**：与普通 SSR 动画一致，不区分硬保底触发
- **异常边界**：SSR 池为空时回退至权重表（理论上不发生活着但容错）
- **落地点**：DrawCardPool.xlsx, DrawCardWeight.xlsx

### R-05: 专武单抽
- **触发条件**：玩家在专武卡池界面点击"抽一次"
- **前置限制**：玩家等级 ≥ 27；持有 ≥ 1 张专武票券（限时 112）或钻石 ≥ 150
- **操作流程**：同角色单抽，产出物为武器
- **状态变化**：武器保底计数器 +1；若 SSR 则归零
- **资源变化**：-150 钻石 / -1 票券 112；+1 武器
- **UI 反馈**：专武抽卡动画 → 展示武器 → 保底更新
- **异常边界**：同 R-01
- **落地点**：DrawCard.xlsx, DrawCardPool.xlsx, DrawCardWeight.xlsx

### R-06: 重复角色转化为英雄碎片奖励
- **触发条件**：抽到的角色已为玩家拥有
- **前置限制**：无
- **操作流程**：查询 HeroExtraReward.xlsx，按角色品质查 FirstReward（首次重复）与 RepeatReward（后续重复）→ 发放对应奖励
- **状态变化**：角色重复计数 +1
- **资源变化**：获得碎片/通用材料等奖励
- **UI 反馈**：抽卡结果界面标注"已拥有"标签 + 显示转化奖励
- **异常边界**：HeroExtraReward 中缺失该角色品质 → 发默认奖励或告警
- **落地点**：HeroExtraReward.xlsx

### R-07: 重复武器转化为武器碎片奖励
- **触发条件**：抽到的武器已为玩家拥有
- **前置限制**：无
- **操作流程**：查询 WeaponDuplicateReward.xlsx → 按武器品质发放对应碎片/材料
- **状态变化**：武器重复计数 +1
- **资源变化**：获得武器碎片/强化材料
- **UI 反馈**：结果界面标注重复 + 显示转化奖励
- **异常边界**：WeaponDuplicateReward 配置缺失 → 发默认碎片
- **落地点**：WeaponDuplicateReward.xlsx

## 技术方案
### 架构影响
- 保底计数器由服务端维护（按池独立存储），客户端仅做展示
- 抽卡结果服务端先行计算，客户端播放动画为纯表现层
- 需支持 DrawSpecialChooseGacha 自选 UP 功能（预留）

### 数据变更
| 表 | 变更类型 | 说明 |
|----|----------|------|
| DrawCard.xlsx | 配置 | 卡池基础信息、票券消耗、解锁条件 |
| DrawCardPool.xlsx | 配置 | 池内物品列表与展示权重 |
| DrawCardWeight.xlsx | 配置 | 稀有度权重阶梯（含软保底区间） |
| DrawSpecialChooseGacha.xlsx | 配置 | 自选 UP 物品配置 |
| HeroExtraReward.xlsx | 配置 | 重复英雄奖励（首次 + 后续） |
| WeaponDuplicateReward.xlsx | 配置 | 重复武器奖励 |

### 接口变更
- `DrawCard` 协议：入参 DrawType(1=单抽/10=十连) + PoolId；出参 DrawResults[] + PityCounter
- `QueryPity` 协议：入参 PoolId；出参 CurrentPity + MaxPity
- 无新增协议，沿用已有 gacha 接口框架

## 风险评估
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 保底计算误差导致提前/延迟保底 | 低 | 高 | 服务端权威计算，日志留底，对账脚本 |
| 权重表配置错误导致出货率崩盘 | 中 | 高 | 上线前采样测试 10 万次验证分布 |
| 重复转化为有价值资源的经济失控 | 中 | 中 | FirstReward/RepeatReward 数值与规划对齐 |
| 断线丢奖 | 低 | 高 | 抽卡结果先写入 db 再扣资源，断线后补发 |
| 前 7 天票券投放过量导致付费意愿下降 | 中 | 中 | 与运营确认总投放量，分期发放 |
