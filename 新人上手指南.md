# Vgame 驾驭工程新人使用手册

> 本文是 Vgame 策划驾驭工程的唯一新人入口，合并了原《新人上手指南》和《驾驭工程使用说明》。适用于第一次接触 Git、SVN、Codex、Skill 和知识图谱的策划。

## 1. 一分钟理解这套工程

驾驭工程不是游戏客户端，也不是另一份配置仓库。它负责把策划规则、设计文档、任务、项目 Skill、真实配置和知识图谱串成可追踪的工作流。

| 位置 | 版本管理 | 权威内容 | 是否允许 AI 默认写入 |
|---|---|---|---|
| `${VGAME_ROOT}` | SVN | 游戏工程、真实 Excel、代码、Unity 资源 | 否，必须明确授权 |
| `${VGAME_DESIGN_ROOT}` | Git | Design、Proposal、Task、Skill、Harness、整合图谱 | 按任务允许 |
| `${VGAME_HARNESS_ROOT}` | Git 启动层 | 中文安装、更新、检查和启动脚本 | 按任务允许 |
| `${VGAME_SOURCE_DOCS_ROOT}` | 本机或共享目录 | 英雄设计、经济框架、战斗框架等原始策划资料 | 默认只读 |
| `${VGAME_OUTPUT_ROOT}` | 本机目录 | AI 生成的报告、量化表、模拟结果 | 允许写入 |

简单记忆：SVN 只管真实项目；Git 统一管理策划知识、Skill、图谱和驾驭工程；Source 是输入资料，Output 是个人生成结果。

## 2. 第一次使用前准备

电脑需要安装：

| 工具 | 用途 | 判断方式 |
|---|---|---|
| Git | 拉取和提交 `Vgame_design` | PowerShell 执行 `git --version` |
| SVN 或 TortoiseSVN | 更新和提交真实项目 | 右键目录能看到 SVN 菜单 |
| Python 3.10+ | 运行图谱、Excel 读取和检查脚本 | 执行 `python --version` |
| Codex | 使用 Vgame 策划 Agent | 能打开本机工作目录 |
| 公司 Git SSH 权限 | 访问公司 Git 仓库 | `git pull` 不出现 publickey 错误 |

`openpyxl` 不需要手工处理，首次安装脚本会自动检查和安装。

## 3. 第一次配置本机路径

先克隆 `Vgame_design` Git 仓库，再进入 `${VGAME_DESIGN_ROOT}\驾驭工程`，双击 `配置本机路径.bat`。

脚本会询问三个目录；设计仓库和驾驭工程目录会根据脚本当前位置自动确定：

| 提示 | 应填写什么 | 是否必填 |
|---|---|---|
| `Vgame SVN root` | 包含 `Config` 和 `HorizonFlyProject` 的 SVN 工程根目录 | 必填 |
| `Source design documents root` | 包含英雄设计、经济框架、战斗框架等源文档的目录 | 按需 |
| `Personal output root` | 角色量化表、审查报告等个人产物目录 | 推荐填写 |

已有正确路径时直接按 Enter，表示保留当前值。出现以下内容表示保存成功：

```text
[DONE] Local paths were saved outside Git and SVN.
```

配置实际保存在：

```text
%LOCALAPPDATA%\Vgame\design-harness.env.bat
${VGAME_DESIGN_ROOT}\local.env.bat
```

这两个文件不会上传。每位成员使用自己的路径，团队不需要统一盘符。

## 4. 第一次安装

在 `${VGAME_DESIGN_ROOT}\驾驭工程` 双击 `一键安装.bat`。它会：

1. 检查 Git、Python 和 `openpyxl`。
2. 确认当前目录属于 `Vgame_design` Git 仓库。
3. 连接本机 Vgame SVN 工作副本。
4. 写入本机路径配置。
5. 检查 Skill、Harness 和知识图谱。

正常结束应看到：

```text
PASS (FAIL: 0, WARN: 0)
Setup complete
```

`FAIL` 必须处理；`WARN` 可以继续阅读原因，但不要直接忽略未知警告。

## 5. 第一次启动

1. 双击 `${VGAME_HARNESS_ROOT}\启动知识图谱.bat`。
2. 浏览器通常打开 `http://127.0.0.1:5173`。
3. 保持启动窗口开启，关闭窗口就会停止 Dashboard。
4. 打开 Codex，选择 `${VGAME_DESIGN_ROOT}` 作为工作目录。
5. 直接用中文说明任务、范围以及是否允许修改真实配置。

Dashboard 只负责可视化。没有打开 Dashboard 时，AI 仍然可以读取仓库中的图谱 JSON。

## 6. 每天开工的固定顺序

1. 右键 `${VGAME_ROOT}` 或本次涉及的小目录，执行 `SVN Update`。
2. 双击 `${VGAME_HARNESS_ROOT}\更新驾驭工程.bat`，更新 Git 策划仓库。
3. 双击 `${VGAME_HARNESS_ROOT}\检查驾驭工程.bat`。
4. 需要可视化时启动知识图谱。
5. 用 Codex 打开 `${VGAME_DESIGN_ROOT}`，开始任务。

如果“更新驾驭工程”提示存在本地修改，不要删除文件。先到 `${VGAME_DESIGN_ROOT}` 执行 `git status`，确认修改属于谁，再完成提交或请负责人处理。

## 7. 中文启动按钮说明

| 文件 | 用途 | 会不会上传内容 |
|---|---|---|
| `配置本机路径.bat` | 设置当前电脑的目录 | 不会 |
| `一键安装.bat` | 首次安装或修复环境 | 不会 |
| `打开驾驭工程.bat` | 打开本地 Git 仓库 | 不会 |
| `更新驾驭工程.bat` | 执行 Git 快进拉取 | 只下载 |
| `检查驾驭工程.bat` | 运行只读门禁和图谱检查 | 不会 |
| `启动知识图谱.bat` | 启动 UA Dashboard | 不会 |

这些按钮都不会替你执行 SVN Commit 或 Git Push。

## 8. 怎么向 AI 描述任务

### 只读分析

```text
分析资源副本金钱本什么时候开放，读取真实配置和代码，不要修改任何文件。
```

### 设计方案

```text
为新的 Boss 挑战建立 Design、Proposal 和 Task，先不修改真实配置。
```

### 明确授权改表

```text
先列出影响范围和待修改行，确认引用闭环后修改指定 Excel；不要修改其他表。
```

### 质量审查

```text
只读检查本次配置改动的断引用、公式列、DropId、Unlock 和 UIlevel 闭环。
```

AI 的正确工作顺序是：先读 `AGENTS.md`，再路由专项 Skill，用图谱缩小范围，最后回到真实 Excel、代码或已确认文档核实。图谱推断不能直接当作最终事实。

## 9. 任务分级和工作流

### 最小路径

以下任务不需要为了流程完整制造多余文档：

- 只读查询、解释和定位。
- 单个数值建议且不落真实表。
- 不影响其他系统的低风险文字修正。

最小路径通常是：任务确认 -> 读取证据 -> 输出结论 -> 标记事实、推断和待确认项。

### 正式功能或高风险改动

跨系统、跨表、涉及经济循环、商业化、战斗平衡或正式配置交付时，使用 Step 0 加九步流程：

| 阶段 | 主要产物 | 完成标准 |
|---|---|---|
| Step 0 项目适配 | Vgame Skill 路由 | 已确认项目事实和权威来源 |
| Step 1 任务登记 | 任务目标、参考资料 | 知道为什么做、参考什么 |
| Step 2 需求契约 | 目标、非目标、范围 | 不再依赖模糊口头理解 |
| Step 3 澄清冻结 | 决策记录、待确认项 | 关键歧义已解决或明确挂起 |
| Step 4 核心设计 | Design Doc | 核心规则、状态和边界完整 |
| Step 5 专项设计 | 数值、经济、战斗、关卡等专项 | 风险模块有独立论证 |
| Step 6 一致性评审 | Proposal、审查结论 | 跨系统引用和规则闭环 |
| Step 7 UI 资产 | UI 流程或资产需求 | 仅在任务需要时执行 |
| Step 8 正式 Excel | 真实配置改动 | 明确授权后才写入 SVN 工程 |
| Step 9 追踪验收 | Task、检查结果、用户验收 | 可复核、可回滚、可交付 |

## 10. Design、Proposal、Task 的区别

| 产物 | 回答的问题 | 存放目录 |
|---|---|---|
| Design Doc | 系统是什么、核心规则如何运作 | `design/` |
| Proposal | 这次准备改什么、影响什么、如何验证 | `proposals/` |
| Task | 谁按哪些步骤执行、当前做到哪里 | `tasks/` |
| Skill | AI 应如何理解项目事实和执行专业流程 | `skills/` |
| Output | 本次分析、量化、模拟的个人结果 | `${VGAME_OUTPUT_ROOT}` |

新设计从 `design/TEMPLATE.md` 开始；需要评审落地时使用 `proposals/TEMPLATE.md`；批准执行后使用 `tasks/TEMPLATE.md`。

## 11. Git 操作详解

Git 只管理 `${VGAME_DESIGN_ROOT}`。

开始工作前，在该目录打开 PowerShell：

```powershell
git status
git pull --ff-only origin master
```

提交时只选择本次确认的文件：

```powershell
git status
git add "具体文件路径"
git diff --cached
git commit -m "docs(guide): update beginner workflow"
git push origin master
```

Git 概念：

- `pull`：下载别人已经推送的内容。
- `add`：选择本次提交文件。
- `commit`：在本机创建版本记录。
- `push`：把本机提交上传到公司 Git。

不要随意使用 `git add .`、`git push --force`、`git reset --hard`。Push 被拒绝或出现冲突时停止操作，先确认远端变化。

## 12. SVN 操作详解

SVN 只管理 `${VGAME_ROOT}` 的真实项目内容。

标准操作：

1. 开工前右键目标目录，执行 `SVN Update`。
2. 修改完成后执行 `SVN Check for modifications`。
3. 新文件先执行 `SVN Add`。
4. 选择 `SVN Commit`。
5. 只勾选本次确认修改的文件。
6. 填写能说明目的和影响范围的提交说明。

SVN 的 Commit 会直接上传服务器，这一点与 Git 不同。不要在 `${VGAME_ROOT}` 根目录不看清单就全部提交，也不要随意执行 Revert。

出现 `Conflicted` 时停止提交。不要擅自选择“使用我的版本”或“使用对方版本”，先由相关人员确认合并方案。

## 13. 同时修改 Git 和 SVN

例如“修改真实掉落表，并补充对应经济 Skill”属于跨仓任务：

1. 先更新 SVN 和 Git。
2. 在 Git 写 Design、Proposal 或 Task，明确影响范围。
3. 获得授权后在 SVN 修改真实配置。
4. 运行配置和引用检查。
5. 先确认 SVN 真实改动可用，再提交 SVN。
6. 在 `CROSS-REPO-CHANGES.md` 登记跨仓关系。
7. 最后提交并推送 Git 文档、Skill 和任务状态。

两边都提交后，任务才算完整交付。

## 14. 知识图谱怎么用

知识图谱由三类来源合并：

```text
客户端代码图 + 注册配置表 Schema + 策划文档索引
                         ↓
                  Vgame 整合知识图谱
```

常用入口：

| 操作 | 入口 |
|---|---|
| 启动可视化 | `${VGAME_HARNESS_ROOT}\启动知识图谱.bat` |
| 更新完整图谱 | `${VGAME_DESIGN_ROOT}\knowledge-graph\更新知识图谱.bat` |
| 检查完整图谱 | `python scripts/check_vgame_graph.py` |
| 更新客户端代码和配置基础图 | `${VGAME_DESIGN_ROOT}\knowledge-graph\更新知识图谱.bat` |

以下情况需要更新完整图谱：

- 新增、删除或重命名配置表。
- 修改字段、表头或 `__tables__.xlsx`。
- 客户端代码结构发生明显变化。
- 新增重要 Design、Proposal、Task 或 Skill。

只改普通数值通常不需要更新结构图谱。`启动知识图谱` 只负责显示，不会重新生成数据。

## 15. 提交前检查

在 `${VGAME_DESIGN_ROOT}` 运行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/check.ps1
powershell -ExecutionPolicy Bypass -File scripts/doc-check.ps1
python scripts/check_vgame_graph.py
```

检查结果含义：

- `PASS`：当前门禁通过。
- `WARN`：可以继续调查，但必须阅读原因。
- `FAIL`：不得提交或宣称完成。

路径门禁会拒绝 Git 受控文件中的个人绝对盘符。个人路径只能写入被忽略的 `local.env.bat`。

## 16. 常见问题

### 更新驾驭工程提示有本地修改

在 `${VGAME_DESIGN_ROOT}` 执行 `git status`。不要删除不认识的修改；确认后提交，或交给修改者处理。

### 出现 `Permission denied (publickey)`

SSH 公钥没有登记到公司 Git，或当前账号没有仓库权限。需要配置 SSH 公钥后再执行安装。

### 图谱页面打不开

确认启动窗口仍在运行，再检查 `http://127.0.0.1:5173`。端口占用时 Dashboard 可能自动选择下一个端口，以窗口显示地址为准。

### AI 找不到源策划 Excel

重新运行 `配置本机路径.bat`，确认 `Source design documents root` 指向实际存放源文档的目录。

### 只修改一个数值也要完整九步吗

不一定。低风险单点数值修改可走最小路径，但仍要确认源表、公式列、引用关系、SVN 差异和验收结果。

### 换电脑或移动目录怎么办

重新双击 `配置本机路径.bat`。不要修改仓库里的 Skill、脚本或 TOML 来适配个人盘符。

## 17. 新人安全红线

1. 未获明确授权，不修改真实 Excel、代码和 Unity 资源。
2. 提交前必须查看 Git 和 SVN 变更清单。
3. 不提交不认识的文件，不还原别人留下的修改。
4. 不把 `local.env.bat`、个人输出和本机缓存加入版本库。
5. 不直接编辑 `server_json` 等生成物替代源 Excel。
6. 不覆盖公式列，不把推断引用当成确认事实。
7. 不使用强制 Push、硬重置或未经确认的冲突覆盖。
8. 有 `FAIL` 时不得交付；有 `WARN` 时必须说明处理结论。

## 18. 每日五分钟清单

```text
[ ] SVN Update
[ ] 更新驾驭工程

```

## 19. 在一个全新对话中怎么使用

新对话不会继承旧对话的聊天记忆，但会读取工作区中的项目入口文件。标准方式：

1. 在 Codex 中新建对话。
2. 工作目录选择 `${VGAME_DESIGN_ROOT}`，不再从 SVN 工程目录启动。
3. 根目录 `AGENTS.md` 会引导 AI 读取本机路径配置、`${VGAME_ROOT}` 和 `vgame-core-understanding`。
4. 第一条消息直接说明任务、范围和读写权限，不需要重新介绍整个项目。

推荐第一句话：

```text
这是 Vgame 项目。只读分析资源副本金钱本的开放、关卡、奖励和经济风险，不修改真实配置。
```

需要正式设计时：

```text
这是 Vgame 项目。为新的 Boss 挑战建立 Design、Proposal 和 Task，按项目总控和专项 Skill 执行，暂不修改真实配置。
```

需要修改真实表时必须明确：

```text
这是 Vgame 项目。先分析并列出拟修改文件、字段和影响链；得到我确认后再修改真实配置。
```

正常情况下，AI 会先确认已加载的项目入口和相关 Skill，再开始读取证据。若 AI 表现得像不了解 Vgame，可粘贴 `${VGAME_HARNESS_ROOT}/新对话提示词.txt` 的内容作为手工兜底。

如果提示找不到设计仓库或 Skill，先退出当前任务，重新运行 `配置本机路径.bat` 和 `一键安装.bat`，然后新建对话。
