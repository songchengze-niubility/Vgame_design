# Vgame 角色套件设计总览

## 定位

`vgame-character-kit-design-map` 负责 Vgame 的角色套件设计理解，不直接替代战斗配置、成长配置、奖励配置或数值平衡。

它回答这些问题：

- 这个角色的战斗身份是什么？
- 普攻、自动技能、必杀、天赋、能效被动、升星、专武之间如何形成闭环？
- 旅券套装、映射装备、通用被动如何放大或补足角色机制？
- 这个设计在正式玩法里应该验证什么：主线推进、Boss、无限挑战、竞技场、爬塔、大秘境，还是资源/装备副本效率？
- 这份设计是否已经有足够证据进入配置、调参或交付审查？

## 与其他 skill 的边界

| 问题 | 主 skill | 本 skill 的角色 |
|---|---|---|
| 角色定位、技能套件、专武/套装/词条关系 | `vgame-character-kit-design-map` | 主责 |
| Hero/Skill/Buff 表、角色属性、突破、升星、专武配置、技能编辑器、runtime 序列化 | `vgame-hero-skill-config-map` | 读取真实角色/技能配置链 |
| 技能、Buff、子弹、召唤、伤害效果的真实配置链 | `vgame-battle-content-map` | 提供设计意图和待验证机制 |
| 升星、专武、旅券、映射装备如何转化为属性/战斗表现 | `vgame-growth-combat-conversion-map` | 提供套件关系和角色语境 |
| DPS、TTK、技能系数、存活、强度曲线 | `vgame-battle-tuning-helper` / `game-numerical-analysis` | 提供待测假设，不做最终数值结论 |
| 源表、字段、Luban、JSON、配置落地 | `vgame-config-schema` | 提供设计源文档，不判断 schema |
| 奖励、掉落、ItemId 产销、开放节奏 | 对应奖励/经济/进度 skill | 只在角色获取或装备来源影响套件时关联 |

## 正式性判断

默认正式性优先级：

1. 用户当前明确指定的文档、sheet、行、角色或机制。
2. `角色技能设计` 中的当前角色块。
3. `专武技能`、`旅券技能`、`映射装备技能`、`通用被动效果` 中的结构化表。
4. 真实配置表与 runtime 实现证据。
5. `角色（留档）` 仅作历史对照。
6. `角色技能备忘录（后期删除）` 仅作实现风险和机制需求记录，不当作正式角色设计。

如果同一角色在设计文档、留档、配置表中冲突，先报告冲突并标记 `待确认`，不要自行选择一个版本当事实。

## Vgame 角色套件的默认理解

Vgame 是 5 人小队、跑酷战斗、自动普攻、自动技能、玩家择机释放必杀的项目。角色套件不是传统站桩卡牌技能组，应默认关注：

- 移动战斗中的命中、覆盖、持续时间和触发窗口。
- 自动技能循环与必杀释放时机的关系。
- 队伍中攻击/防御/辅助角色的节奏互补。
- Boss、长线 endurance、PVPVE 竞速、爬塔单队 DPS、大秘境环境适配等验证场景差异。
- 子弹、护盾、召唤、幻象、Buff、控制、治疗、能量、冷却、DOT、破挡等机制是否能被 runtime 支持。

## 后续维护规则

新增角色设计文档、战斗设计文档或表格导出时：

1. 先记录文件路径、sheet 名、有效区域、字段结构。
2. 区分正式设计、留档、草稿、备忘录。
3. 记录导出 Excel 的读取坑点，例如合并单元格、公式引用、维度元数据错误、重复 ID。
4. 不把未验证的设计文本直接写成配置表事实。
5. 如发现新增源表或真实配置表，转到 `vgame-config-schema` 的新表接入规则。
