# Vgame Design Rules

## 设计文档什么时候需要写

以下情况必须写 `design/` 文档：

- 新增玩法、系统、角色定位、养成线或资源循环。
- 修改跨多个配置表的规则。
- 修改影响商业化、留存、付费、战斗验证或版本节奏的设计。
- 反推历史系统并希望长期沉淀。

简单字段修正、错别字、单表小改可以直接进入 Task，但仍要写清影响范围。

## 设计文档必须回答的问题

| 问题 | 说明 |
|---|---|
| 目标 | 为什么要做，解决哪个体验或数值问题 |
| 玩家路径 | 玩家在哪里看到、何时进入、怎样完成 |
| 配置链 | 涉及哪些表和关键字段 |
| 经济链 | 产出什么、消耗什么、是否冲击存量系统 |
| 战斗链 | 是否影响属性、技能、关卡、Boss、PVP/PVE |
| UI/开放 | 是否涉及 UIlevel、Unlock、SystemOpen、引导 |
| 风险 | 误配、膨胀、断档、体验噪点、上线成本 |
| 验收 | 怎样证明设计成立 |

## 推荐结构

```markdown
# 系统名 Design Doc

## 1. 背景
## 2. 目标
## 3. 玩家路径
## 4. 规则设计
## 5. 配置链
## 6. 经济与战斗影响
## 7. UI 与开放
## 8. 风险与缓解
## 9. 验收标准
```

模板见 `design/TEMPLATE.md`。
